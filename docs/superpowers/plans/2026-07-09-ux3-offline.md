# UX-3 — Offline genérico Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development. Steps use checkbox (`- [ ]`) syntax.

**Goal:** Cola offline genérica multi-entidad (registro + militante + respuesta) con blobs, para que la captura en campo no se pierda sin señal.

**Architecture:** Generalizar `frontend/src/offline/` de registro-only a un job-queue tipado (kind + payload + blobs). Handlers por kind al drenar; blobs subidos tras el create idempotente. Integrar en las 3 capturas. + dedup backend de responses.

**Tech Stack:** TS/React/idb (IndexedDB), vitest; Python/FastAPI/pytest.

## Global Constraints

- Idempotencia por `client_uuid` en TODO create (registro/militante ya; response se añade en T1). Replay nunca duplica.
- El drenado conserva la lógica actual: red (status undefined) → `error` reintentable; 4xx no-transitorio → `failed` terminal; recuperar filas `syncing` colgadas.
- Blobs viven en IndexedDB; se **borran al sincronizar** (dato sensible INE/firma).
- Migración IndexedDB v1→v2 idempotente (migra `registro_queue`→`job_queue`).
- Mantener `CapturaPage` funcionando (wrapper `enqueue` o migrarlo en T7).
- Build limpio + vitest verde + pytest verde. Frontend desde `frontend/`, backend desde `backend/`.

_db = await openDB<AgoraDBSchema>("agora-offline", 2, {
  async upgrade(db, oldVersion, _n, tx) {
    if (!db.objectStoreNames.contains("job_queue")) {
      const s = db.createObjectStore("job_queue", { keyPath: "client_uuid" });
      s.createIndex("by_status", "status"); s.createIndex("by_created_at", "created_at");
    }
    if (oldVersion < 2 && db.objectStoreNames.contains("registro_queue")) {
      const old = await tx.objectStore("registro_queue").getAll();
      const jobs = tx.objectStore("job_queue");
      for (const r of old) await jobs.put({ ...r, kind: "registro", blobs: [] });
      db.deleteObjectStore("registro_queue");
    }
  },
});

- [ ] **Step 3: Implement** — reescribir `drainQueue` para despachar por `job.kind` al handler correspondiente (mezcla `deps.handlers` sobre los defaults). Default handlers:
```ts
import { createRegistro } from "@/api/registros";
import { createMilitante, uploadDocumento } from "@/api/militantes";
import { submitResponse } from "@/api/atencion";

const DEFAULT_HANDLERS: Record<JobKind, JobHandler> = {
  registro: async (job) => { await createRegistro(job.payload as any); },
  militante: async (job) => {
    const m = job.server_id
      ? { id: job.server_id }
      : await createMilitante(job.payload as any);        // idempotente por client_uuid
    // persistir server_id ANTES de subir fotos, para que un fallo de subida no re-cree
    await markStatus(job.client_uuid, "syncing", { server_id: m.id });
    for (const b of job.blobs) {
      await uploadDocumento(m.id, b.slot as "frente"|"reverso"|"firma", b.data);
    }
  },
  response: async (job) => {
    if (!job.server_id) await submitResponse(job.payload as any);   // idempotente por client_uuid
    // (subir evidencia si el flujo lo requiere — usar el endpoint de evidencia existente)
  },
};
```
Conservar la clasificación red/permanente y la recuperación de `syncing`. Al éxito total del handler → `removeQueued` (los blobs se van con la fila → borrado del dato sensible). Mantener el guard `draining`.
